# 技术栈与架构说明

## 1. 项目目标

智泳云枢要解决的不是单一模型推理问题，而是“游泳训练视频分析产品化”的问题。系统需要覆盖：

- 视频上传与文件管理
- 分析任务创建与状态追踪
- 重模型推理调用
- 可视化工作台展示
- HTML 报告生成
- 后续对象存储、任务队列、PDF 导出的升级空间

## 2. 总体架构

```text
┌───────────────────────────────────────────────┐
│ Vue3 + TypeScript + Vite 前端                  │
│ Element Plus / ECharts / Pinia / Canvas        │
└───────────────────────┬───────────────────────┘
                        │ REST API
┌───────────────────────▼───────────────────────┐
│ 业务后端 FastAPI                                │
│ 上传、任务、结果、报告、状态管理                 │
└───────────────┬───────────────────┬───────────┘
                │                   │
                ▼                   ▼
          MySQL 数据库          本地 uploads
                │
                ▼
┌───────────────────────────────────────────────┐
│ 独立模型服务 FastAPI                            │
│ YOLO / MMPose / OpenCV / FFmpeg / PyTorch      │
└───────────────────────────────────────────────┘
```

## 3. 技术栈选择

### 3.1 前端

- `Vue3`：适合中后台和分析平台场景，组件组合灵活
- `TypeScript`：约束 API 数据结构和复杂页面状态
- `Vite`：开发启动快，适合拆分独立前端工程
- `Element Plus`：快速搭建任务流、表单、表格、状态组件
- `ECharts`：适合指标趋势图、雷达图、对比图
- `Pinia`：轻量状态管理
- `Axios`：统一封装业务 API
- 原生 `video`：第一版先满足播放需求
- `Canvas`：绘制关键点、骨架线、轨迹、检测框和阶段叠加层

### 3.2 业务后端

- `FastAPI`：Python 栈友好，接口定义和调试成本低
- `Pydantic`：输入输出结构校验
- `SQLAlchemy`：数据库模型与关系映射
- `MySQL`：主业务数据持久化
- `BackgroundTasks`：MVP 阶段的异步任务执行器

### 3.3 模型服务

- `FastAPI` 独立服务：隔离重依赖和 GPU 运行环境
- 后续模型依赖：`YOLO`、`MMPose`、`OpenCV`、`FFmpeg`、`PyTorch`

### 3.4 文件和报告

- 文件存储：本地 `uploads/` 起步
- 升级方向：MinIO
- 报告输出：HTML 报告先行，PDF 后置

## 4. 服务职责划分

### 前端负责

- 视频上传交互
- 训练元数据录入
- 任务列表和任务状态展示
- 工作台视频播放和 Canvas 叠加
- 图表与报告页渲染

### 业务后端负责

- 上传视频保存
- 视频、任务、结果、报告元数据入库
- 调用模型服务
- 保存结构化分析结果
- 对前端暴露统一业务 API

### 模型服务负责

- 接收视频分析请求
- 执行检测、关键点、阶段识别、指标计算
- 返回统一结构化结果
- 不直接承担用户、任务、权限、数据库管理职责

## 5. 当前实现状态

- `frontend-vue/` 已具备上传页、任务页、工作台、报告页
- `backend/` 已具备上传、任务、结果、报告 API
- `model_service/` 已提供 stub 分析 API
- `frontend/` 仍保留为早期 Next/React 原型参考
