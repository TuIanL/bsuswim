## Context

项目当前已有游泳分析平台的前端原型和 OpenSpec 规格，能力集中在页面导航、视频分析任务流、可视化工作台和互动报告。下一阶段需要从前端演示走向真实视频上传、重模型分析、结果存储和报告生成。

模型侧会使用类似 YOLO 的重模型，可能依赖 GPU、深度学习框架、视频编解码库和较大的模型权重。业务后端不应直接承载这些依赖，否则会造成部署复杂、启动慢、资源竞争和故障边界不清晰。因此本设计将业务后端和模型服务拆成两个 FastAPI 服务。

目标运行形态：

```text
┌───────────────────────────────────────────────┐
│ Vue3 + TypeScript + Vite 前端                  │
│ Element Plus / ECharts / video / Canvas        │
└───────────────────────┬───────────────────────┘
                        │ REST API
┌───────────────────────▼───────────────────────┐
│ 业务后端 FastAPI                                │
│ Pydantic / SQLAlchemy / MySQL / BackgroundTasks │
│ 上传、任务、结果、报告、权限边界                 │
└───────────────┬───────────────────┬───────────┘
                │                   │
                ▼                   ▼
          MySQL 数据库          本地 uploads
                │                   │
┌───────────────▼───────────────────▼───────────┐
│ 独立模型服务 FastAPI                            │
│ YOLO 类模型 / 视频抽帧 / 姿态或动作分析 / 指标计算 │
└───────────────────────────────────────────────┘
```

## Goals / Non-Goals

**Goals:**

- 固化前端、业务后端、数据库、文件存储、模型服务和异步任务的技术栈。
- 支持用户上传游泳视频并创建真实分析任务。
- 支持业务后端以异步方式调用独立模型服务，避免 HTTP 请求等待完整模型推理。
- 支持任务状态持久化，前端可以轮询或刷新后恢复任务状态。
- 支持保存模型分析结果，驱动视频叠加、指标图表和 HTML 报告页。
- 为后期迁移到 MinIO、Celery + Redis、PDF 导出和 GPU 部署保留边界。

**Non-Goals:**

- 不在本次变更中实现具体 YOLO 模型训练、标注数据集管理或模型调优。
- 不在本次变更中强制引入 Celery、Redis、MinIO 或 PDF 导出。
- 不在本次变更中设计多租户计费、细粒度权限或生产级模型编排平台。
- 不要求模型服务直接访问业务数据库；模型服务通过 API 输入和输出结构化结果。

## Decisions

### 1. 前端采用 Vue3 + TypeScript + Vite

决策：平台正式实现采用 Vue3、TypeScript、Vite、Element Plus、ECharts、Pinia 和 Axios。视频播放优先使用原生 `video`，复杂控制需求出现后再引入 Video.js。骨架、检测框、角度线和轨迹叠加使用 Canvas。

理由：该组合适合数据分析型后台产品，开发速度快，Element Plus 能覆盖表单、表格、状态、弹窗和布局，ECharts 适合指标趋势和对比图。Canvas 比 DOM 元素更适合高频渲染关键点和轨迹层。

替代方案：继续沿用现有 Next/React 原型。它适合展示页和概念验证，但用户已明确希望确定 Vue3 技术栈，因此后续实现应将现有交互和视觉经验迁移到 Vue 平台。

### 2. 业务后端采用 FastAPI + SQLAlchemy + Pydantic + MySQL

决策：业务后端负责上传、任务、元数据、结果、报告和对模型服务的调度。数据校验使用 Pydantic，ORM 使用 SQLAlchemy，主数据库使用 MySQL。

理由：FastAPI 与模型生态同属 Python 技术栈，便于共享 schemas、类型和部署经验。MySQL 适合团队常见部署环境，任务状态、视频记录、报告元数据和结果索引都属于关系型数据。

替代方案：使用 Node.js 后端或直接由模型服务承担业务 API。前者会增加模型集成语言边界，后者会让业务依赖和 GPU 推理依赖互相污染。

### 3. 模型服务独立部署

决策：YOLO 类模型、视频抽帧、检测、姿态估计、动作阶段识别和指标计算放在独立 FastAPI 模型服务中。业务后端通过内部 REST API 调用模型服务，传入视频文件引用和任务参数，接收结构化分析结果。

理由：重模型服务可能需要 GPU、CUDA、PyTorch、OpenCV、FFmpeg 和大模型权重，部署频率、资源需求和故障模式都不同于业务后端。独立服务可以单独扩容、单独重启、单独迁移到 GPU 机器。

替代方案：在业务后端进程里直接加载模型。MVP 看似简单，但会带来启动慢、内存占用高、并发阻塞和依赖冲突问题，不适合作为主路径。

### 4. BackgroundTasks 起步，但任务状态必须入库

决策：MVP 使用 FastAPI `BackgroundTasks` 调度分析执行，但 `analysis_tasks` 表必须记录任务状态、进度、错误、输入、输出和时间戳。前端通过 API 查询任务状态，而不是依赖内存状态。

任务状态建议：

```text
uploaded -> queued -> processing -> result_saving -> completed
                         │
                         └───────────────> failed
```

理由：BackgroundTasks 可以降低起步复杂度；任务状态入库则保证服务重启后仍能展示任务历史，并为 Celery + Redis 迁移提供连续的数据契约。

替代方案：一开始引入 Celery + Redis。它更适合生产并发，但会增加部署和调试成本。当前阶段可以先保留任务表和任务处理接口，后期替换执行器。

### 5. 文件存储本地 uploads 起步，抽象存储接口

决策：MVP 将上传视频保存到本地 `uploads`，数据库保存文件路径、原始文件名、大小、MIME type 和 checksum。代码层保留 `StorageService` 或等价边界，后期可切换 MinIO object key。

理由：本地文件系统最适合早期开发和演示；但视频文件天然会变大，后期需要对象存储。提前避免在业务逻辑里散落绝对路径，可以降低迁移成本。

替代方案：立即使用 MinIO。它更接近生产，但会让本地开发多一个基础设施依赖。当前更适合把对象存储作为后期升级项。

### 6. HTML 报告页优先，PDF 导出后置

决策：报告能力先以 HTML 页面交付，直接读取分析结果生成摘要、诊断、指标图表和训练建议。PDF 导出作为后期能力，可以基于同一份报告数据生成。

理由：HTML 报告页更易迭代、可交互、可嵌入视频片段和图表。PDF 导出涉及分页、字体、图表渲染和异步生成，适合在数据结构稳定后再做。

替代方案：先做 PDF。它适合交付归档，但会拖慢早期报告结构验证。

## Risks / Trade-offs

- 模型推理时间长导致任务堆积 → MVP 限制并发、记录任务状态和错误；后期迁移 Celery + Redis 并增加队列监控。
- 本地 uploads 在多人部署或容器重启时不稳定 → 开发期明确为本地方案；生产前迁移 MinIO，并使用 object key 替代本地路径作为业务引用。
- 业务后端和模型服务的结果 schema 不稳定 → 使用 Pydantic 定义版本化结果结构，模型服务返回 `schema_version`。
- 前端从现有 Next/React 原型迁移到 Vue3 有成本 → 先迁移业务流程和组件行为，再复用视觉设计语言，不要求逐文件照搬。
- Canvas 叠加与视频时间轴同步存在精度问题 → 前端以视频 `currentTime` 为基准，根据帧时间戳匹配关键点和检测框；缺失数据时显示 unavailable 状态。
- BackgroundTasks 不适合生产级长任务可靠性 → 只作为起步执行器，任务表和 API 设计要保持可迁移。

## Migration Plan

1. 保留现有前端原型作为交互参考，建立新的 Vue3 + Vite 应用骨架。
2. 建立业务后端 FastAPI 项目，连接 MySQL，并创建视频、任务、结果和报告相关数据模型。
3. 实现本地 `uploads` 存储服务和上传 API。
4. 实现分析任务创建、状态查询、任务列表和报告数据 API。
5. 建立独立模型服务 FastAPI 项目，先提供 stub 推理结果，再接入 YOLO 类模型。
6. 业务后端通过 BackgroundTasks 调用模型服务，并将结果保存到数据库。
7. 前端接入真实 API，替换本地 demo job client。
8. 当任务量、视频体积或部署形态需要时，将本地 uploads 迁移到 MinIO，将 BackgroundTasks 迁移到 Celery + Redis。

Rollback 策略：在真实后端未稳定前，前端保留 demo 数据模式；模型服务异常时任务进入 `failed` 并保存错误原因，不影响任务列表和报告页面加载其他任务。

## Open Questions

- 第一版 YOLO 类模型的输入输出格式是否已经确定，尤其是检测框、关键点、阶段标签和置信度字段。
- 是否需要用户登录和权限隔离；如果需要，应作为单独 change 处理。
- MySQL 部署方式是本机、Docker Compose 还是远程数据库。
- 第一版视频大小、时长和格式限制需要定到什么范围。
- 模型服务是否必须支持 GPU 环境自动检测，还是先由部署文档约束。
